# Projection: Explicit Viewmodels Over Elmish State

Projection is the explicit CLR-facing path. Use it when the Avalonia-facing
contract is named viewmodel classes with properties, methods, and
collection adapters.

The F# model owns state transitions. The projection owns the object shape
that AXAML sees.

```fsharp
type FormModel =
    { Name: string
      Newsletter: bool }

type Msg =
    | SetName of string
    | SetNewsletter of bool
```

The projection owns the mutable object Avalonia binds to.

```csharp
public sealed class FormProjection : ObservableObject
{
    public string Name { get; private set; } = "";

    public void Update(FormModel model)
    {
        Name = model.Name;
        OnPropertyChanged(nameof(Name));
    }
}
```

## Snapshot-Host Projection For Shallow Viewmodels

For small screens, a projection can expose one immutable snapshot through
`SnapshotHost<'T>`.

```csharp
public sealed class FormHost : SnapshotHost<FormView>
{
    public FormHost(FormView initial) : base(initial) {}

    public void Submit() => Dispatch(Msg.SubmitForm);
}
```

AXAML can bind through the current snapshot.

```xml
<TextBlock Text="{Binding Current.Name}" />
```

This keeps a named CLR host while avoiding a deep tree of mutable properties.

Use this when a conventional host object helps but most displayed values
come straight from an immutable F# view snapshot.

## Classic Projection Trees For Mutable Control Identity

Some Avalonia controls benefit from stable mutable row objects. A projection
tree keeps those objects and updates them when Elmish produces a new model.

```csharp
Rows.SyncWith(
    model.Rows,
    row => row.Id,
    vm => vm.Id,
    row => new RowProjection(row),
    (vm, row) => vm.Update(row));
```

This suits selection-sensitive lists, virtualization-sensitive lists, and rows
with local command surfaces.

The tradeoff is more CLR viewmodel code in exchange for explicit object
identity and command surfaces.

## What Changes In A Typical Feature

Projection usually touches more files than ElmView:

- the F# model and update logic
- the projection class or row adapter
- the AXAML binding surface

That cost buys an explicit CLR contract that C# and Avalonia tooling can see
directly.

## When Projection Is A Good Fit

Use Projection when a screen needs named CLR types as part of its design.

It also fits screens where row identity, selection, or view-specific commands
matter more than minimizing boilerplate.

Use ElmView when the F# record is the main UI schema and the CLR surface is
binding glue.
