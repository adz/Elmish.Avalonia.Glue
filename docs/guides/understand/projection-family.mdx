# Projection family

The **Projection** family is for teams that prefer explicit, named CLR viewmodels (C# or F# classes). It provides the most control over the Avalonia-facing contract and is ideal for complex, command-heavy screens.

## Two sub-styles

The Projection family supports two primary ways of authoring viewmodels:

### 1. Snapshot-Host Projection (Lightweight)

This is the simplest way to use named viewmodels. You create a shallow class that inherits from `SnapshotHost<'T>`. It exposes the current immutable snapshot and can also expose explicit commands that dispatch messages.

- **Best for**: Read-mostly screens or simple forms where you still want a named class for better Avalonia tooling support or C# consumption.
- **How it works**: You bind your AXAML to properties on the `SnapshotHost` (e.g., `DataContext.Current.UserName`).

### 2. Classic Projection Tree (Deep)

In this style, you build a tree of mutable viewmodel classes that implement `IProjection<'Model>`. Each viewmodel is responsible for updating itself when a new model snapshot arrives.

- **Best for**: Selection-sensitive views, large complex lists, or views that require significant logic in the property getters (e.g., complex formatting or calculated fields).
- **How it works**: When the top-level model updates, you propagate the update down the viewmodel tree using `SyncWith` extensions for collections.

## Core interfaces

- `IProjection<'Model>`: Defines an `Update` method that accepts the new model snapshot.
- `IDispatchTarget<'Msg>`: Defines a `SetDispatch` method to connect the viewmodel to the Elmish loop.

## What you can do

- **Explicit CLR contract**: Use named properties and methods that are easy to discover via IntelliSense and friendly to compiled bindings.
- **Customized row identity**: Maintain stable row viewmodels in lists even as the underlying data changes, which is critical for preserving selection and scroll state.
- **Bridge C# and F#**: Authored F# model/update logic can be easily consumed by C# UI teams through these familiar viewmodel patterns.
