# Shared substrate

`Elmish.Glue.Core` provides the foundational building blocks for both authoring families. Its primary goal is to bridge the gap between immutable Elmish state and the mutable, notification-heavy world of Avalonia.

## The problem: Stability vs. Immutability

Avalonia (and XAML in general) expects stable object identities that notify when properties change. Elmish, however, works by producing entirely new state snapshots.

If we simply replaced the `DataContext` with a new record on every update:
1. Existing bindings would often break or require expensive re-evaluation.
2. Control state (like scroll position, focus, or selection) might be lost.
3. Animations and transitions would be interrupted.

The shared substrate solves this by providing **stable bindable shells** that wrap **changing immutable snapshots**.

## Core concepts

### BindableNode

The base class for everything that needs to notify the UI. It implements `INotifyPropertyChanged` and provides a protected `NotifyPropertyChanged` method.

### BindableSnapshotHost

This is the "shell" for a top-level view. It holds a single `Snapshot` property and notifies the UI whenever a new snapshot is provided via the `Update` method.

- **Stability**: The host instance itself remains the same for the lifetime of the view.
- **Identity**: It uses `obj.ReferenceEquals` to skip updates if the snapshot hasn't actually changed.

### BindableSnapshotNode

For nested data, we don't always want to expose the entire root snapshot. `BindableSnapshotNode` allows you to create a stable, bindable object for a *part* of the snapshot tree.

- **Projection**: It takes a function to "pluck" its specific data from the root snapshot.
- **Propagation**: When the root host updates, it triggers a `RefreshSubtree` call that propagates down the node graph, raising `PropertyChanged` for any properties exposed by the nodes.

### KeyedSnapshotCollection

Lists are particularly sensitive to identity. If you replace a `List` or `ObservableCollection` entirely, Avalonia has to throw away all the UI controls (rows) and recreate them.

`KeyedSnapshotCollection` uses a **patching algorithm** to sync an `ObservableCollection` with an immutable list:
- It uses keys to identify which items were added, removed, or moved.
- It updates existing items in place if they are already present, preserving the stable UI container.

## What you can do

- **Hold a stable dispatch surface**: Keep a consistent `dispatch` function available to the UI even as the model changes.
- **Refresh without rebuilding**: Update the entire UI tree efficiently using identity-aware snapshot propagation.
- **Patch collection identity**: Maintain smooth scrolling and selection in large lists by patching rather than replacing.
