# ElmView family

**ElmView** is the F#-first path. It is designed to minimize boilerplate by using plain F# records as your primary UI schema. Instead of writing manual projection classes, you bind your AXAML to a generated (or mechanically authored) host facade that wraps your immutable records.

## The ElmView V2 Philosophy

ElmView V2 focuses on making interactive views feel like "normal" Avalonia authoring while keeping the state purely functional.

### 1. F# Records as UI Schema
Your view model is a hierarchy of plain, immutable F# records. They represent exactly what you want to see on the screen.

```fsharp
type UserInputView = {
    Name: string
    Email: string
    Newsletter: bool
}
```

### 2. Normal AXAML Bindings
You use ordinary Avalonia `Binding` syntax in your AXAML. You don't need custom markup extensions or special control libraries.

- `Mode=OneWay`: Binds to a snapshot property for display.
- `Mode=TwoWay`: Binds to an editable property. The host facade automatically dispatches the mapped Elmish message when the user changes the value.

### 3. Generated Host Facade
The `GeneratedViewHost` (and nested `GeneratedViewNode`s) provides the bindable surface that Avalonia expects. It implements `INotifyPropertyChanged` and handles the mapping between UI property changes and Elmish messages.

### 4. Centralized Write-Back Mappings
Instead of scattering message dispatch logic through your AXAML or code-behind, you register all editable fields in one place near host construction.

```csharp
bindings.For(x => x.UserInput.Name).Dispatch(Msg.NewSetName);
```

## How it works

1. **Snapshots**: When the Elmish model updates, the host receives a new `AppView` snapshot.
2. **Propagation**: The host calls `RefreshSubtree()`, which triggers `PropertyChanged` events across the entire generated node graph.
3. **Write-Back**: When a `TwoWay` binding updates a property on the host, the host looks up the registered mapping and calls `dispatch`.

## What you can do

- **Minimize boilerplate**: No need to write deep trees of manual projection classes for every screen.
- **Ordinary AXAML**: Keep your UI code clean and readable, using standard Avalonia features like Compiled Bindings.
- **Design-time parity**: Use the same host shape and snapshots to drive the Avalonia previewer, ensuring what you see in the designer is what you get at runtime.
