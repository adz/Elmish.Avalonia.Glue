# ElmView: F# Records With An Avalonia Binding Surface

ElmView is the F#-first path. Screen-shaped data is a plain immutable F#
record. Avalonia binds to CLR properties exposed over that record.

ElmView makes records like this the authored UI schema:

```fsharp
type UserInputView =
    { Name: string
      Email: string
      Newsletter: bool
      ValidationText: string }
```

That nested record is normally part of a root view record:

```fsharp
type AppView =
    { UserInput: UserInputView }
```

The AXAML path uses the root record field name, not the nested record type
name. `UserInputView` is the type. `UserInput` is the field that becomes the
first segment in `{Binding UserInput.Name}`.

The CLR host exists because Avalonia needs a bindable object with properties
and `PropertyChanged`.

Today, this repository provides the host and node base classes, and the sample
host is handwritten in the shape a generator would produce. There is not yet a
source generator or CLI command that turns F# records into host code. When the
docs say "generated host" or "generated node", read that as "code with the
generated/mechanical shape"; in the current samples, that code is authored by
hand.

## What Avalonia Sees

AXAML uses standard bindings. There is no custom markup extension and no
custom control set.

```xml
<TextBox Text="{Binding UserInput.Name, Mode=TwoWay}" />
<CheckBox IsChecked="{Binding UserInput.Newsletter, Mode=TwoWay}" />
<TextBlock Text="{Binding UserInput.ValidationText}" />
```

`OneWay` display bindings read from the current immutable snapshot. `TwoWay`
editable bindings call a setter on the generated-shaped node.

The binding path must exist on the host or one of its nodes. If AXAML says
`UserInput.Name`, the host needs a `UserInput` property and that object needs a
`Name` property.

In the sample, the handwritten host exposes the root segment:

```csharp
private readonly UserInputNode _userInput;

public UserInputNode UserInput => _userInput;
```

The node then plucks the nested F# record from the latest root snapshot:

```csharp
public UserInputNode(AppHost host)
    : base(
        () => host.View,
        host.Dispatch,
        host.RegisterChildNode,
        view => view.UserInput,
        new[] { nameof(Name), nameof(Email), nameof(Newsletter) })
{
}
```

That is the full path:

```text
AppView.UserInput field
-> AppHost.UserInput CLR property
-> UserInputNode.Name CLR property
-> {Binding UserInput.Name}
```

That shape is mechanical. Application meaning belongs in the F# view, AXAML,
and write-back mapping.

## What The Generated Node Does

A generated-shaped or mechanically authored node reads from the latest snapshot.
Writable properties do not mutate the snapshot. They dispatch messages.

```csharp
public string Name
{
    get => Snapshot.Name;
    set => Host.TryDispatchWriteBack("UserInput.Name", value);
}
```

When Elmish produces a new snapshot, the host raises `PropertyChanged` across
the node graph so Avalonia refreshes its bindings.

The setter path goes back to Elmish. It does not mutate the snapshot in place;
the next snapshot comes from `update`.

## Where Message Routing Lives

Write-back routing is centralized near host construction.

```csharp
bindings.For(x => x.UserInput.Name).Dispatch(Msg.NewSetName);
bindings.For(x => x.UserInput.Email).Dispatch(Msg.NewSetEmail);
bindings.For(x => x.UserInput.Newsletter).Dispatch(Msg.NewSetNewsletter);
```

Interaction meaning stays out of AXAML and out of F# record metadata. The
binding path names the editable value. The mapping names the Elmish message.

`bindings.For(...)` has one job: register a write-back route.

```csharp
bindings.For(x => x.UserInput.Name)
        .Dispatch(value => Msg.NewSetName(value));
```

The selector is converted to the path string `"UserInput.Name"`. Later, when a
node setter calls `TryDispatchWriteBack("UserInput.Name", value)`, the
host finds that route and dispatches `Msg.NewSetName(value)`.

If a property is display-only, it does not need a write-back route.

```xml
<TextBlock Text="{Binding UserInput.ValidationText}" />
```

If a property is editable, it needs both the AXAML `TwoWay` binding and the
write-back route.

```xml
<TextBox Text="{Binding UserInput.Name, Mode=TwoWay}" />
```

```csharp
bindings.For(x => x.UserInput.Name).Dispatch(Msg.NewSetName);
```

## What Kinds Of Values Can Be Bound

ElmView can expose values that Avalonia can bind to through CLR properties.

Common display bindings:

- `string`, `int`, `float`, `bool`, and other scalar values
- computed text such as `ValidationText` or `FormatLabel`
- nested records exposed as generated child nodes
- lists exposed as `IReadOnlyList<T>` or collection properties
- styling values such as brushes, colors, flags, and indexes

Common editable bindings:

- `TextBox.Text` to `string`
- `CheckBox.IsChecked` to `bool`
- `ComboBox.SelectedItem` to a selected value
- `ComboBox.SelectedIndex` to `int`
- `Slider.Value` or similar range controls to numeric values
- multiline `TextBox.Text` to `string`

Editable bindings map to messages that represent user intent. The
generated setter dispatches the message; the Elmish `update` function decides
how the immutable model changes.

## What Changes In A Typical Feature

For a new editable field, change:

- add a field to the F# view record
- render it in AXAML with binding syntax
- ensure the host exposes the matching CLR property or node
- add one write-back mapping from path to message if the field is editable

The host code stays mechanical. Today the sample host is handwritten in that
generated shape; the intended direction is to generate it rather than author it
by hand.

Feature reviews usually touch the F# model, F# view record, design snapshot,
AXAML, and editable-path message routes.

## When ElmView Is A Good Fit

Use ElmView when the F# view type is the source of truth for what the screen
can display.

It fits forms, dashboards, and pages where most UI state is derived from
immutable model data. It is weaker for controls with bespoke mutable behavior
or heavy row-level state.

Use Projection when the explicit CLR viewmodel is part of the design rather
than only infrastructure.
